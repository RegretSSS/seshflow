## 🎯 关于子任务功能的说明

### 为什么接受子任务功能？

**1. 原有设计**
- 代码库中已有完整的子任务逻辑：
  - `task-manager.js` 中有 `addSubtask` 和 `toggleSubtask` 方法
  - `import.js` 中有子任务解析代码
  - 数据结构中有 `subtasks` 数组

**2. 实际需求**
- 任务分解是项目管理的核心需求
- 子任务作为检查点帮助跟踪进度
- AI 可以根据子任务提供更精准的帮助

**3. 不是过度设计**
- 功能已存在，只是有 bug
- 修复 bug 不增加复杂性
- 不增加用户学习成本

**4. 使用场景**
```markdown
- [ ] 实现用户认证 [P0] [auth] [6h]
  - [x] 设计数据库表 [2h]
  - [ ] 实现登录接口 [3h]
  - [ ] 编写单元测试 [1h]
```

这不是过度设计，而是：
- ✅ 任务分解的最佳实践
- ✅ 进度跟踪的有效方式
- ✅ AI 辅助的必要信息

### 子任务 vs edit/delete/move 命令

| 功能 | 决策 | 理由 |
|------|------|------|
| 子任务 | ✅ 接受 | 原有功能，修复 bug，不增加命令 |
| edit 命令 | ❌ 拒绝 | 新增命令，增加学习成本 |
| delete 命令 | ❌ 拒绝 | 新增命令，增加学习成本 |
| move 命令 | ❌ 拒绝 | 新增命令，增加学习成本 |

**关键区别**：
- 子任务是**修复现有功能**
- edit/delete/move 是**新增命令**

## 🔍 详细分析

### ✅ 接受的建议

#### 1. 修复子任务功能 ⭐

**问题描述**:
- 缩进的子任务被解析为独立任务
- 显示任务时不显示子任务

**根本原因**:
```javascript
// Bug: trim 导致缩进信息丢失
const line = lines[i].trim();
// 下面的检查永远不会匹配
if (line.startsWith('  -') || line.startsWith('\t-')) {
```

**修复方案**:
```javascript
// 保留原始行以检测缩进
const rawLine = lines[i];
const line = rawLine.trim();

// 使用原始行检测子任务
if (rawLine.startsWith('  -') || rawLine.startsWith('\t-')) {
```

**修复效果**:

**修复前** (8 个独立任务):
```
✓ Would import 8 tasks:
1. 主任务 1
2. 子任务 1.1    ← 独立任务
3. 子任务 1.2    ← 独立任务
4. 子任务 1.3    ← 独立任务
...
```

**修复后** (2 个主任务 + 子任务):
```
✓ Would import 2 tasks:
1. 主任务 1
   Subtasks: 3

2. 主任务 2
   Subtasks: 3
```

**显示效果**:
```
┌─ 主任务 1
├───────┐
│ Subtasks (1/3):
│   ✅ 子任务 1.1 [2h]
│   ⏸️ 子任务 1.2 [1h]
│   ⏸️ 子任务 1.3 [1h]
└───────┘
```

**为什么不是过度设计**:
- 功能已存在，只是修复 bug
- 不增加新的命令
- 不增加用户学习成本
- Markdown 格式简单直观

#### 2. 统一数据类型

**问题描述**:
- `actualHours` 字段既有字符串又有数字
- `estimatedHours` 字段可能为字符串

**影响**:
- 统计计算可能出错
- JSON 数据不一致
- AI 解析时可能出错

**修复方案**:
1. 修改 `task-manager.js` 中的 `completeTask` 函数
2. 修改 `task-manager.js` 中的 `createTask` 函数
3. 创建数据修复脚本 `fix-data-types.js`
4. 运行脚本修复现有数据

**效果**:
- ✅ 所有 actualHours 统一为数字
- ✅ 所有 estimatedHours 统一为数字
- ✅ 统计功能正常工作
- ✅ JSON 输出类型一致

---

#### 3. 优化统计计算

**问题描述**:
- `actualSpentHours` 计算可能有误

**修复方案**:

**效果**:

- ✅ 显示 8.8h 实际工时

### ❌ 拒绝的建议

#### 1. 添加编辑功能

**建议内容**:
```bash
seshflow edit <task-id> --title "新标题"
seshflow edit <task-id> --priority P1
```

**拒绝理由**:
1. **增加使用难度**:
   - 需要记住多个参数
   - 需要查找命令语法
   - 比 JSON 编辑更复杂

2. **不必要**:
   - JSON 文件可以直接编辑
   - 编辑器提供语法高亮
   - 编辑速度更快

3. **实现成本高**:
   - 需要实现多个子命令
   - 需要参数验证
   - 需要错误处理

**替代方案**:
```bash
# 直接编辑 JSON
code .seshflow/tasks.json
```
